# Design: API Client Refactor (Orval + Next.js App-Specific Fetching)

**Status:** Implemented  
**Spec:** `spec.md` (same folder)

---

## 1. Design Goals

1. Keep API access strongly typed from OpenAPI contract to feature usage.
2. Keep API package framework-agnostic and simple.
3. Standardize error/result handling in one place.
4. Preserve existing architecture boundaries (`app/`, `features/`, `lib/`, `packages/*`).

---

## 2. High-Level Architecture

### 2.1 Layers

1. **Contract Layer** (`packages/api-client/src/generated/*`)
   - Generated by Orval from backend OpenAPI spec.
   - Contains endpoint functions and schema types.
2. **Core Layer** (`packages/api-client/src/core/*`)
   - Common fetch mutator, normalized errors, and result helpers.
3. **Domain API Layer** (`packages/api-client/src/admin/*`, `src/public/*`)
   - Small domain modules composing generated endpoints into app-facing APIs.
4. **App Orchestration Layer** (`apps/*/src/features/**/services|server`, `apps/*/src/lib/api/*`)
    - App-specific concerns like cookies, session, cache, query invalidation, and UI state.
    - In `apps/admin`, server-side auth translation from HttpOnly access-token cookies to backend `Authorization: Bearer` headers lives at this boundary.

### 2.2 Deliberate Non-Pattern

- No generic `Repository<T>` abstraction.
- No inheritance hierarchy for endpoint modules.
- API organization is explicit by backend domain and app use-case.

## 4. Orval Generation Strategy

1. Source: `.specs/admin/backend-openspec-api.json`
2. Output mode: split by tags/domains for manageable generated files.
3. Client style: fetch-based generation.
4. Use a shared mutator/fetch wrapper to normalize request and error behavior.
5. Keep generated artifacts under `src/generated` and avoid hand edits.

---

## 5. Data Flow by App

### 5.1 `apps/admin` (TanStack Query First)

```text
Client Component
  -> feature hook (TanStack Query)
  -> feature service (same-origin call or orchestration)
  -> app/api route handler (thin)
  -> feature server/lib api adapter
  -> packages/api-client (admin domain module)
  -> backend API
```

Rules:

- Query keys, invalidation, and optimistic updates stay in `features/*/hooks`.
- API route handlers remain thin and delegate to server handlers.
- Feature components never call backend endpoints directly.
- Protected backend admin calls derive bearer auth server-side from the HttpOnly access-token cookie before calling the backend API.
- Refresh flows use backend body-token exchange (`{ refreshToken }`) and avoid forwarding auth cookies/bearer to the backend.

### 5.2 `apps/web` (Server-First)

```text
Server Component / server action / route handler
  -> feature server service
  -> packages/api-client (public domain module)
  -> backend API
```

Rules:

- Server-side fetching is default.
- Use Next.js caching/revalidation at the server boundary.
- TanStack Query appears only in isolated interactive client islands.

---

## 6. Error and Result Contract

1. Keep one normalized error type exported from `packages/api-client`.
2. Keep one normalized success/error result union for transport calls.
3. Preserve trace-id extraction and backend metadata propagation.
4. Remove duplicated payload/error parsing where equivalent helpers already exist.

---

## 7. Migration Strategy

1. Introduce Orval generation and core package structure first.
2. Keep compatibility exports for existing admin consumers.
3. Migrate one vertical slice first (`auth`, `episodes`) and verify behavior.
4. Migrate remaining admin domains incrementally.
5. Document `apps/web` usage pattern and apply it during web implementation.

---

## 8. Validation Strategy

1. Contract generation check in CI (`generate` + `contract:check`).
2. `@cafedebug/api-client` lint/typecheck pass.
3. `@cafedebug/admin` lint/typecheck/tests pass after migration.
4. Manual smoke checks for login + episodes list/detail/mutation flows.
5. Confirm no architecture violations (thin routes, no client-side backend fetch).
